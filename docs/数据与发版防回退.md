# 数据与发版防回退（死灰不复燃）

你掌握全部代码与库时，**表里的内容完全听 SQL 的**：谁最后把一份「整表/整库」灌进去，长什么样就什么样。要防止「侧栏/菜单又变回老样子」，关健不是改包名，而是**关死错误的数据入口** + **能一眼发现**。

## 1. 唯一真源

- 应用**只连**库名 **`yibo`**（`DRUID_MASTER_URL` 等）。不要在生产保留第二个「官方种子库」还指向应用。
- **初始化**：仅使用仓库内 **`backend/RuoYi-Vue/sql/yibo_init_baseline.sql` + 如需调度 `quartz.sql`**；**已上线的库不要**再对整库重跑基线（会 `DROP` 表，等于推倒重来）。
- **增量**：以 **`database/scripts/`** 中脚本为准，尤其完整链 **`install_all_business_menus.sh ... yibo`**。

## 2. 禁止/慎用

- **不要**将不明来源的 `mysqldump` 全量导入生产 `yibo`（除非走专项评审与演练）。
- **不要**在仓库中提交**大体积**、易覆盖 `sys_menu` 的 `database/dumps/*.sql`（已 `.gitignore`）。
- 恢复前想清楚：这份备份里的 `sys_menu` 是否仍是你要的。

## 3. 发版/跑完业务链之后（门禁）

- `install_all_business_menus.sh` **末步**会自动跑 **`verify_yibo_menu_sanity.sql`**，非 `PASS` 时脚本**非 0 退出**（可设 `YIBO_SKIP_MENU_VERIFY=1` 仅作调试，勿用于「假装成功发版」）。
- 可单独跑：`./scripts/verify-yibo-menu-sanity.sh yibo`（与发版/恢复同连接方式、同密码）。
- 全仓库**漂移体检**（配置里是否又写回**旧库名**、误留 dump 等）：`./scripts/yibo-drift-audit.sh`。

## 4. 若仍出现「和预期不符的菜单」

1. 确认当前服务连的库：日志或配置里的 `jdbc` 是否 **`yibo`**。  
2. 对 `yibo` 跑 **`verify-yibo-menu-sanity.sh`** 与业务链。  
3. 非超管角色：看 **`sys_role_menu`** 是否仍挂旧 `menu_id`。

## 5. 心理模型（一句）

**复燃 = 又执行了一份「整份」旧数据文件，或连错了库，不是 MySQL 故意和你作对。**
